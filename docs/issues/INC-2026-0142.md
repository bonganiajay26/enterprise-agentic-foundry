## Description

`payment-service` pods entered `CrashLoopBackOff` in production at 14:32 UTC,
~4 minutes after a routine Helm values rollout (release `payment-service-1.42.0`).
12 of 12 replicas were affected; the service was fully unavailable for 6 minutes
until traffic was automatically shifted to the previous ArgoCD revision.
Estimated impact: ~340 failed checkout attempts, all retried successfully
post-rollback (no data loss; Stripe idempotency keys prevented double-charges).

## Evidence

- **Logs:** `{service="payment-service", level="error"} |= "issue_id=INC-2026-0142"` →
  `FATAL: missing required environment variable DB_CONNECTION_POOL_SIZE`
- **Metrics:** Grafana "Platform / SLOs" → payment-service availability dropped
  to 0% for 6 minutes (panel: `slo:payment_service:availability:30m`)
- **Traces:** No traces emitted — service failed at startup, before tracer init
- **Alerts fired:** `KubePodCrashLooping` (14:34 UTC), `SLOErrorBudgetBurn` (14:36 UTC)
- **Screenshots / artifacts:** `s3://compliance-evidence/incidents/2026-06/INC-2026-0142/`

## Root Cause

- Symptom: Pods crash-looping immediately after rollout
- Why? → Container fails at startup with a missing env var error
- Why? → `DB_CONNECTION_POOL_SIZE` is required by the new connection-pooling
  code merged in `payment-service` v1.42.0
- Why? → The Helm `values-prod.yaml` change that should have added this env
  var was in a *separate* PR that hadn't merged yet
- Why? → The release checklist doesn't cross-reference application PRs against
  their corresponding Helm values changes before tagging a release
- **Root cause:** No automated gate validates that a service's required
  configuration (env vars / secrets) is present in the target environment's
  Helm values *before* the release is promoted — this is a release-process gap,
  not a code defect.

## Resolution

`remediation_agent` classified this as `category=helm_fix`, `environment=production`
→ classification **AUTOMATIC_WITH_CANARY** (helm fixes in prod always canary first).

1. Merged the pending `values-prod.yaml` PR adding `DB_CONNECTION_POOL_SIZE: "25"`
2. Re-synced via ArgoCD to the corrected revision
3. Canaried to 10% of replicas, validated startup + smoke tests, then completed rollout
4. Re-ran `tests/smoke/smoke-test.sh production` — all green

## Prevention

- **New CI gate** (`PR #1843`): every release now fails fast if a service's
  declared required configuration isn't present in the target Helm values —
  this is the structural fix; it makes this entire class of incident impossible.
- **New regression test**: `tests/integration/k8s-health.py::test_required_env_vars_present`
  validates all 12 production services on every deploy.
- **Runbook updated**: `runbooks/pod-crashloop.md` now documents the
  "missing required config after release" pattern with a one-line diagnostic command.
